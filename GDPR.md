# GDPR

If you need strict, user-specific consent data linked directly to your database, you can use the open-source django-gdpr-cookie-consent package on your Django backend. This allows you to manage consent, record it, and handle cookie policies in a unified manner across both frontends.

**Important Considerations for Your Setup:**

- **Cookie Domain:** Since you have separate frontend/app domains, ensure your cookies are set properly for the parent domain (e.g., `yoursite.com`) to avoid third-party cookie blocking issues.
- **Backend Auth:** If using Django's session authentication with a separate Next.js app, ensure `SESSION_COOKIE_DOMAIN` and CORS are configured properly in `settings.py`.

## Right to Portability

The easiest way to meet the GDPR "Right to Portability" requirement is to create a dedicated API endpoint in your **Django DRF** backend that aggregates a user's personal data into a structured, standard format like JSON or CSV and provides it as a downloadable file. Needs a double opt-in workflow.

**Build a Self-Service Export Endpoint**

Instead of handling requests manually, implement a "Download My Data" button in your **Next.js** app settings. This button should trigger a GET request to a secured Django REST Framework (DRF) view.

- **Authentication:** Use `IsAuthenticated` permissions to ensure users can only download their own data.
- **Response Format:** Use `Content-Disposition: attachment` in the HTTP response headers to force the browser to download the file rather than displaying raw text.
- **Standard Formats:** GDPR requires a "machine-readable" format; JSON is the standard for web apps, but providing a CSV option is highly recommended for non-technical users.

**Streamline with Existing Tools**

You can avoid building the export logic from scratch by using packages designed for Django data management:

- **django-import-export:** This is the most popular tool for quickly defining "Resources" (models/fields) to be exported. It integrates with both the Django admin and custom views to generate Excel, CSV, or JSON files.
- **django-gdpr-assist:** A specialized tool that helps you specifically track and export Personal Identifiable Information (PII). You define which fields on which models are private, and it provides tools to export or anonymize them.

**What to Include in the Export**

To be fully compliant, your export should include more than just profile details:

- **Core Profile:** Name, email, and account settings.
- **Activity Logs:** Interaction history or any data generated by the user within the app.
- **Consent History:** A log of when they accepted or changed their cookie and privacy preferences.
- **Third-Party References:** If you share data with providers like Stripe or SendGrid, include a list of these partners in the export so the user knows where else their data resides.

**Comparison of Export Methods**

| Method | Implementation Effort | Best For |
| --- | --- | --- |
| **Manual DRF View** | Medium | Simple apps with few data models. |
| **django-import-export** | Low | Fast implementation with multiple format options (Excel, CSV). |
| **django-gdpr-assist** | Medium | Strict compliance needs and managing PII across many models. |

## Right to Erasure

To comply with the GDPR **"Right to Erasure"** (or "Right to be Forgotten"), your setup must ensure that a user can request the permanent removal of their personal data from your database and all associated systems. Needs a double opt-in workflow.

**Hard Deletion vs. Anonymization**

GDPR allows for two primary ways to "forget" a user:]

- **Hard Delete:** Completely remove the user record and all associated data (e.g., activity logs, profiles) from the database. This is the simplest approach but can break database integrity if other records depend on that user.
- **Anonymization:** Replace personally identifiable information (PII)—like names, emails, and IP addresses—with randomized strings (e.g., `user_deleted_123`). This preserves your app's aggregate statistics and database relationships without retaining any identifiable data.

**Implementation with Django and DRF**

Since you are already using a self-service export endpoint, you can mirror that logic for deletion.

- **Custom Delete View:** Create a secure Django REST Framework (DRF) endpoint (e.g., `DELETE /api/user/me/`) that performs the erasure. Ensure you use IsAuthenticated permissions to prevent unauthorized deletions.
- **Leverage `django-gdpr-assist`:** You can use the `django-gdpr-assist` package to automate this. It allows you to define a `PrivacyMeta` class on your models to specify which fields should be deleted or anonymized automatically when a request is made.
- **Handling Backups:** GDPR does not require you to instantly scrub offline backups, but you must keep a record of deletion requests. If you ever restore a backup, you must "replay" those deletions to ensure the "forgotten" users are removed again.

**Frontend Workflow (Next.js)**

Your Next.js marketing and app frontends should provide a clear "Delete My Account" button.

- **Verification:** It is a best practice to require the user to re-enter their password or confirm via a checkbox before sending the `DELETE` request to your backend.
- **Post-Deletion:** Once the backend returns a successful response (e.g., `204 No Content`), clear all local storage, cookies (including cookie consent status), and redirect the user to a "Goodbye" page.

### Summary Table of Actions

| Component | Action |
| --- | --- |
| **User Request** | Triggered via a button in the **Next.js** settings panel. |
| **Data Erasure** | **Django** view either hard deletes the user or anonymizes PII. |
| **Audit Trail** | Maintain a secure log of *when* a deletion happened (without saving the user's data). |
| **Third Parties** | Notify integrated services (like Stripe or Mailchimp) to delete the user's data on their end. |

## Deleting Users with Active Subscriptions

Handling a GDPR "Right to Erasure" request for a paid user requires balancing your privacy obligations against legal financial requirements and your contract with the customer. Under GDPR, the right to be forgotten is **not absolute** and can be overridden by other legal duties.

**Retention for Legal Obligations**

You are **required by law** to retain certain data related to financial transactions and taxes, typically for 7 years depending on your jurisdiction.

- **What to keep:** Invoices, payment records, and basic identifying info needed for tax audits.
- **What to delete:** All non-essential data, including usage logs, personalized settings, marketing preferences, and profile data not required for financial record-keeping.
- **Compliance Tip:** Inform the user clearly that while their account is being "forgotten," specific financial records must be kept to satisfy tax and anti-money laundering laws.

**Handling Remaining Subscription Time**

If a user has paid for a period they haven't used yet, you generally have two paths:

- **Path A: Immediate Erasure & Refund:**
  - Manually or via API, calculate the prorated remaining balance.
  - Issue a refund through your payment processor (e.g., using the Stripe Refund API).
  - Proceed with immediate data erasure.
- **Path B: Scheduled Erasure:**
  - Set the subscription to cancel at the end of the current billing period using `cancel_at_period_end`.
  - Mark the account for deletion. **Restrict access** or disable processing of their data in the interim, but keep the record until the period they paid for expires.
  - Once the subscription naturally ends, trigger your "forget me" script.

*Technical Strategy (Django + Stripe)*

To manage this in your **Django** backend, your deletion logic should check for active subscriptions:

| Step | Action |
| --- | --- |
| **Check Subscription**  | Query your database or Stripe to see if `subscription.status == 'active'`. |
| **User Choice** | In your **Next.js** frontend, ask the user if they want an immediate refund (and instant loss of access) or to keep access until the period ends. |
| **Webhook Logic** | Use a Stripe Webhook for `customer.subscription.deleted`. When it fires, trigger your anonymization/erasure function. |
| **Third-Party Cleanup** | Notify third-party processors like **Stripe** or **Mailchimp** to delete the user's data on their end. |
